# Collider Pipeline Architecture
## Simulation Workflow in Q-Collider

Q-Collider implements a staged collider emulation pipeline inspired by standard high-energy physics simulation workflows. The purpose of the pipeline is to translate theoretical particle models into predicted collider signatures and analysis outputs.

The architecture follows the general sequence used in modern collider studies:

MadGraph → Pythia → Delphes → Analysis

This structure is widely adopted in phenomenological research because it separates event generation, particle showering, detector response modeling, and physics analysis into modular stages.

---

## Pipeline Stages

### Stage 1 — Event Generation (MadGraph)

The first stage produces hard-scattering events from a specified particle interaction model.

Inputs include:

- theoretical model definitions
- particle masses and couplings
- collider beam energy
- parton distribution functions
- event generation parameters

Outputs include:

- parton-level event records
- matrix element calculations
- event files typically stored in LHE format

These outputs represent the fundamental collision interactions before hadronization and detector effects.

---

### Stage 2 — Particle Shower Simulation (Pythia)

The Pythia stage models parton showering and hadronization processes that occur after the initial collision.

This stage simulates:

- QCD parton showers
- hadron formation
- particle decays
- final-state radiation
- underlying event effects

Outputs typically include:

- particle-level event data
- HepMC format event files

These files represent a more realistic particle cascade following the hard interaction.

---

### Stage 3 — Detector Response Approximation (Delphes)

The Delphes stage approximates the response of a collider detector.

This stage models:

- calorimeter responses
- tracking detectors
- jet reconstruction
- missing transverse energy
- trigger effects

Outputs typically include:

- ROOT-format datasets
- reconstructed particle objects
- event-level observables

The goal is to convert particle-level events into detector-level measurements similar to those observed in experiments.

---

### Stage 4 — Physics Analysis

The final stage performs statistical and event-level analysis.

Typical outputs include:

- signal event counts
- background estimates
- invariant mass distributions
- statistical significance estimates
- anomaly indicators

These outputs are stored as structured JSON or analysis result records inside the Q-Collider system.

---

## Pipeline Execution Model

Each pipeline run is represented by a **PipelineRuns** record, which tracks the status of all four stages.

Fields typically include:

- model identifier
- stage status
- file paths for intermediate outputs
- timestamps for stage transitions
- analysis result locations

This structure allows the system to track reproducibility and run history.

---

## Pipeline Modes

Q-Collider currently supports two operational modes.

### Demo Mode

Used for interface validation and rapid experimentation.

Characteristics:

- simulated stage completion
- placeholder output files
- synthetic metrics

This mode allows fast testing of the research workflow.

---

### Research Mode

The intended long-term configuration integrates real simulation tools.

Expected behavior:

- generation of real MadGraph input cards
- execution of Pythia shower simulations
- Delphes detector modeling
- real analysis output

The architecture has been designed so that the simulated stages can be replaced with real toolchain execution without changing the user interface.

---

## Pipeline Outputs

Key data products generated by the pipeline include:

- LHE event files
- HepMC particle records
- Delphes ROOT datasets
- analysis JSON summaries

These outputs form the basis for downstream discovery ranking and validation scoring.

---

## Pipeline Integration with Discovery Map

Once analysis results are produced, the system updates model metadata including:

- signal strength
- validation score
- discovery probability

These updates allow the model to move within the Collider Discovery Map and reflect newly computed information.

---

## Future Extensions

Future versions of the pipeline may include:

- detector-specific configurations
- background modeling workflows
- real dataset comparison tools
- machine learning anomaly detection

The modular design ensures that each stage can be upgraded independently.

---

## Scientific Context

The MadGraph → Pythia → Delphes workflow has become standard in collider phenomenology. By structuring the pipeline around these stages, Q-Collider aligns with established research practices while enabling automated theory exploration.
